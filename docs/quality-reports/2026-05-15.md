## Recommended Priorities

1. **(High — persists)** `src/lib/activity-log.ts:27` — single insert discards Supabase error. Fix: `const { error } = await supabase.from(...).insert({...}); if (error) console.error(...)`.
2. **(High — new)** `src/app/c/[clientId]/s/[shipmentId]/page.tsx:365` — bare `await supabase.from('shipments').update(...)` in customer-facing portal. If the update fails, `receipt_completed_at` is never persisted but the UI shows it as complete until next reload.
3. **(High — persists)** `src/app/shipments/[id]/page.tsx:429` — bare `await supabase.from('trees').update(...)` on individual cancel. Tree may silently stay as `shipped` in DB if the update fails.
4. **(High — new)** `src/app/shipments/[id]/page.tsx:487` — bare `await supabase.from('estimates').update({status:'発行済'})` in cancel-ship flow. Estimate may silently remain as `出荷済` even after shipment is cancelled.
5. **(Medium — persists)** `src/components/ShipmentDialog.tsx:142` — estimate status update on ship is fire-and-forget.
6. **(Medium — persists)** `src/app/shipments/[id]/page.tsx:252` — `const { count }` missing error; if the count query fails, `count` is `null` and the delete branch is skipped silently.
7. **(Medium)** Fix the 14 `const { data: ... }` patterns in 5a — add `error` destructuring so failures surface in production logs. Highest impact: the 3 customer-facing portal queries in `c/[clientId]/page.tsx` and the 3 in `c/[clientId]/s/[shipmentId]/page.tsx`.
8. **(Low)** Type the three `any` usages in `tree-repository.ts` (lines 31, 80, 679) using the Supabase-inferred row type.
