## Check 5: Unhandled Async Calls

2 actionable issues found.

### 5-A · `src/lib/activity-log.ts:27` — fire-and-forget insert (no error capture)

```ts
// logActivity() — line 22-38
await supabase.from('activity_logs').insert({
    tree_id: treeId,
    action,
    details: details || null,
    actor,
})
// return value never captured → { error } silently dropped
```

The function is wrapped in `try/catch`, which only catches thrown JS exceptions. Supabase RPC errors (RLS policy violations, constraint errors, quota) are returned inside the response object and never thrown — they are silently swallowed here. The `logActivityBulk` sibling function on line 56 correctly does `const { error } = await ...`.

**Fix:** Change to `const { error } = await supabase.from('activity_logs').insert({...})` and add `if (error) throw error`.

### 5-B · `src/app/estimates/[id]/page.tsx:203-205` — `.then()` without checking response `error`

```ts
supabase.from('clients').select('id, name, default_rate').order('name').then(
    ({ data }) => { setClients(data || []) },       // ← error field ignored
    (err) => { console.error('[estimates] ...', err) }  // only catches network throws
)
```

The fulfillment callback destructures `{ data }` but ignores `{ error }`. Supabase DB errors (permission denied, network timeout surfaced as Supabase error objects) would leave `clients` state empty with no user feedback. The rejection callback only fires for true JS exceptions, not Supabase-level errors.

**Fix:** Change to `({ data, error }) => { if (error) { console.error(...); return } setClients(data || []) }`.
