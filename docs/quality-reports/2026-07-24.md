## 🆕 New Findings (Medium-Low)

### [NEW-1] `get_shipped_by_fiscal_year()` が `anon` に `execute` 権限を付与

`supabase/migrations/20260708_shipped_by_fiscal_year_rpc.sql` で追加された RPC が `anon` ロールに実行権限を付与している。

```sql
grant execute on function public.get_shipped_by_fiscal_year() to anon;
```

**検証結果: 現状は問題なし（要監視）**

- 関数は `SECURITY INVOKER` で定義されており、呼び出し元の RLS ポリシーが適用される
- `trees` テーブルは全行 `TO authenticated` ポリシー（`20260127_enable_rls.sql` で設定）のため、`anon` が呼び出しても 0件が返る
- ただし将来的に `anon` 向け RLS ポリシーが追加された場合、このグラントが集計データを漏洩させるリスクがある

**推奨**: 必要なければ `revoke execute on function public.get_shipped_by_fiscal_year() from anon;` を追加マイグレーションで当てること。dashboard は `createClient()` (server, authenticated) 経由で呼び出しているため機能は損なわれない。

**Severity: Low**

### [5] Supabase エラーチェック漏れ（4件残存）

**5-1: `src/lib/activity-log.ts` line 27** — `logActivity()` が `supabase.insert()` の戻り値を非チェック。Supabase はエラーを throw しないため try/catch では捕捉できない。ログ失敗がサイレントになる。

```typescript
// 現状: エラーが捨てられる
await supabase.from('activity_logs').insert({ ... })

// 推奨: { error } を受け取って console.error する（logActivityBulk と同じパターン）
const { error } = await supabase.from('activity_logs').insert({ ... })
if (error) throw error
```

**5-2, 5-3, 5-4: `src/app/c/[clientId]/s/[shipmentId]/page.tsx`** — クライアントポータルのデータ取得3箇所で `error` を非チェック。DB エラーが「データなし」扱いになる。

**5-5（🔶 Partial）: `src/hooks/useTrees.ts` line 88–103** — 初回 useEffect の Promise.all に `.catch()` が追加されて改善されたが、`refreshData` コールバック（呼び出し元が catch しない場合は unhandled rejection）はまだ try/catch なし。

### [SEC-1] API ラベルルートの認証チェック漏れ — 前回から変化なし

`src/app/api/label/[id]/route.ts` — `auth.getUser()` チェックなし。tree ID を知っている者なら誰でも `.lbx` ファイルを取得可能。ラベル内容自体の機密度は低いが、QR コードの内容（tree UUID）が確定できる。

### [SEC-A] ログアウト時の IndexedDB クリア処理なし — 前回から変化なし

- Supabase の `signOut()` 呼び出しがコードベースに存在しない（Email OTP ログインのみ、明示的ログアウト機能未実装）
- クライアントポータルの「ログアウト」（`useStaffPin.ts:37` の `logout`）は `localStorage.removeItem()` のみで、Dexie DB クリアなし
- 端末の引き継ぎ・紛失時に IndexedDB の樹木データ・顧客情報・pendingEdits が残留する

Fix: `db.delete()` を logout フロー・signOut フローに追加。または将来ログアウト機能を実装する際に必ずセットで入れること。
