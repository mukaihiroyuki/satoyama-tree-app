## Check 5 — Unhandled async / missing Supabase error checks

Supabase client methods never throw — they return `{ data, error }`. A bare `try/catch` does **not** catch Supabase-level errors. The issues below silently discard errors, risking invisible data-loss or stale UI state.

### 5-1 · `src/lib/activity-log.ts` — line 27

```typescript
await supabase.from('activity_logs').insert({ ... })
```

The `insert()` result is never destructured. If the insert fails (e.g., RLS rejection, network timeout), the error is silently swallowed because `supabase` does not throw — it returns `{ data: null, error: PostgrestError }`. The surrounding `try/catch` only catches JS exceptions, not Supabase error payloads.

**Applies to both `logActivity` (line 27) and `logActivityBulk` (same pattern).**

Fix: `const { error } = await supabase.from(…).insert(…); if (error) console.error(…)`

---

### 5-2 · `src/app/c/[clientId]/s/[shipmentId]/page.tsx` — line 51

```typescript
const { data: clientData } = await supabase
    .from('clients')
    .select('id, name, portal_show_price, portal_password')
    .eq('id', clientId)
    .single()
```

`error` is not destructured. A DB error is indistinguishable from a genuinely missing record — both reach the `if (!clientData)` null-check and display the same "not found" page, masking the real cause.

### 5-4 · `src/app/c/[clientId]/s/[shipmentId]/page.tsx` — line 101

```typescript
const { data: receipts } = await supabase
    .from('client_receipts')
    .select('shipment_item_id')
    .eq('client_id', clientId)
    .in('shipment_item_id', itemIds.length > 0 ? itemIds : ['__none__'])
```

An error here is silently treated as an empty receipt list (`receipts || []`), so a DB error makes every item appear as un-received.

### 5-5 · `src/hooks/useTrees.ts` — line 95

```typescript
const [treesData, speciesData, count] = await Promise.all([
    repo.getAllTrees(onRefresh),
    repo.getAllSpecies(onSpeciesRefresh3),
    repo.getPendingEditCount(),
])
```

This `await Promise.all(...)` inside the `refreshData` `useCallback` has no `try/catch`. A rejection propagates to the caller unhandled, potentially crashing silently. The initial-load `Promise.all` at line 23 and the online-recovery block at line 62 are both properly guarded; this one is not.

Fix: wrap lines 95–102 in `try { … } catch (err) { console.error('refreshData error:', err) }`.
