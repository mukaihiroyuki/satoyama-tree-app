### 1. オフライン編集の全実装（メインタスク）

PWAの「見た目だけオフライン対応」を「データもオフラインで読み書き」に昇格させた。

**アーキテクチャ:**
```
UI (pages)
  ↓
Custom Hooks (useTree, useTrees, useOnlineStatus)
  ↓
TreeRepository (サービスクラス)
  ├→ Dexie.js (IndexedDB) … ローカルキャッシュ＋オフライン書込み
  └→ Supabase Client       … リモート読み書き
```

- オンライン時: Supabaseから取得 → IndexedDBにキャッシュ → UI表示
- オフライン時: IndexedDBから表示 → 編集はローカル保存 → 復帰時に自動同期

### 2. QRスキャンページの検索修正

**問題:** 検索ボタンを押しても何も反応しない
**原因:** `html5-qrcode` と `@/lib/db` の静的importがページ全体のJSを壊していた
**解決:**
- 両モジュールを動的 `import()` に変更
- `<form onSubmit>` → `<button onClick>` に変更
- 検索ロジックも改善: `.single()` → `.ilike()` 部分一致 + `tree_number` フォールバック
- 入力欄が黒背景で見えなかった問題も修正（`bg-white border-2 border-green-400` 追加）

### 3. PWAオフラインキャッシュ強化

**問題:** 機内モードにすると「データにアクセスするには機内モードをオフに」とOS警告が出る
**解決:** `next.config.ts` に以下を追加:
- `cacheOnFrontEndNav: true` — フロントエンドナビゲーション時にキャッシュ
- `aggressiveFrontEndNavCaching: true` — 積極的キャッシュ
- `fallbacks: { document: "/~offline" }` — 未キャッシュページ用フォールバック
- `reloadOnOnline: false` — 復帰時の自動リロード抑止

### 4. 樹木登録フォーム改善

紙台帳の実運用フローに合わせて修正:
- **本立ち**: `[1,2,3,5]` → `[1,2,3,4,5]`（4が抜けていた）＋6本以上は手入力
- **上代**: 6ボタン → 1万〜10万の10ボタン＋10万超は手入力＋リアルタイム円表示

## 学んだこと

### 技術的知見

1. **静的importの罠**: PWAでDexie.jsやhtml5-qrcodeを静的importすると、Service Worker環境やSSRでモジュール解決が失敗し、ページ全体のJSが死ぬ。動的 `import()` が必須。

2. **`navigator.onLine` の限界**: `true` でも実際には電波が弱くて通信できないケースがある。Supabase呼出が応答なしでハングする。ただし今回のユーザーテストでは「読み込み中...が出るが十分速い」とのことで実用上問題なし。

3. **ESLint `react-hooks/set-state-in-effect`**: useEffectの中でuseCallbackを呼ぶとこのルールに引っかかる。`.then()` パターン＋キャンセルフラグで回避。

4. **PWAの `aggressiveFrontEndNavCaching`**: `@ducanh2912/next-pwa` のこのオプションが、ページ遷移時のService Workerキャッシュを劇的に改善する。

### 業務的知見

- 樹木の値付けは現場で実物を見て行う。事前にラベル印刷して貼る運用は現実的でない。
- 紙台帳のフォーマットを実際に見せてもらうことで、フォームの入力項目が正しく設計できた。
- ユーザーが「サクサク動いた」と報告 — 電波の悪い実環境でのオフラインキャッシュが有効に機能している。
